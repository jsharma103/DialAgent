# v0.5 — Dental shopping concierge

The first concrete distribution shape for DialAgent. Narrows Phase 7
to a single vertical and a zero-PII task surface, distributed as an
**MCP server** that Claude / Gemini / OpenAI hosts can invoke — so it
ships to real users without onboarding friction, PHI handling, or
commitment risk.

---

## The wedge

A patient is shopping for a new dentist. They have to call 5-10 offices
to verify the basics (does this office take my insurance, what's the
cash price for a cleaning, do they have Thursday afternoon slots). Total
cost today: 1-3 hours of phone time. Provider directory data online is
famously stale.

DialAgent reduces this to "submit a list of offices, get a summary table
back." That's the v0.5 product.

This is **not** a degraded version of "AI medical agent." It's the
*shopping phase* of medical care — pre-relationship, zero-PII, info
retrieval only. The care-management phase (eligibility checks,
scheduling, refills) is v1+ and requires the full PHI/auth/Mode-2 stack.

---

## Distribution shape (MCP-first)

The primary surface is an **MCP server**, not our own chat UI. Building
yet another chat to compete with Claude.ai / ChatGPT / Gemini / Cursor
is meaningless. Expose DialAgent as a tool those hosts can invoke.

```
User (in Claude Desktop / Claude.ai / ChatGPT / Cursor / Zed / etc.)
  ↓ "find a dentist near 94110 that takes Delta Dental PPO"
LLM host  →  DialAgent MCP server  →  FastAPI app  →  Twilio
                                                       ↓
                              transcript + structured answer
                              bubbles back to the chat
```

- The LLM host does NL parsing, conversation, and result rendering.
- DialAgent just does the phone call + extraction.
- The web form (already built at `localhost:8000`) is a secondary
  mobile / demo surface.

**Initial MCP tools** (mapped onto existing FastAPI endpoints):

| Tool | What it does |
|---|---|
| `place_call(phone, task)` | Starts a call. Returns `call_sid` immediately. |
| `get_call_status(call_sid)` | Polls for live transcript / final structured result. |
| `parse_provider_pdf(path)` | (Shape 2 / v0.6) Extract `{name, phone}` entries from a PDF. |
| `batch_call(entries, task)` | (Shape 2 / v0.6) Orchestrate N sequential calls. |
| `get_batch_status(batch_id)` | (Shape 2 / v0.6) Per-call state + aggregate. |

**Deployment**:
- v0.5: local **stdio** MCP server. Users install via JSON snippet in
  Claude Desktop / Cursor / etc. Runs alongside the FastAPI app.
- v0.6+: hosted **HTTP+SSE** MCP server for non-technical users.

**Monetization (deferred until there's actual demand)**:
- **BYOK forever-free**: open-source repo, user plugs in their own
  Twilio + Deepgram + Anthropic keys.
- **Hosted PAYG**: ~$0.30/min via Stripe-metered credits. Added once
  the free tier gets exercised. Keep total perceived cost below
  ~$0.50/min on top of the user's existing Claude subscription —
  anything higher feels insane.
- Real per-call cost is ~$0.04/min (Twilio + Deepgram + Anthropic).
- Don't monetize before there are users hitting the free tier hard.

**Caveat — MCP-native paid is early-mover territory.** Most monetized
MCP servers today (Stripe, Apify, Browserbase, Vapi, E2B, Tavily) are
existing paid SaaS that also expose MCP. DialAgent would be among the
first MCP-native consumer concierges. Real opportunity, fewer
playbooks to copy.

## In scope

- **One vertical**: dental.
- **Tier 1 tasks only** (info-only, zero PII required):
  - Plan acceptance check ("do you take Delta Dental PPO?")
  - Pricing ("what's the cash price for a cleaning?")
  - Hours / location / parking
  - Procedure availability ("do you do implants? sedation?")
  - In-network status verification from the provider side
- **Web form** → call → live transcript view → results page
- **Click-to-dial fallback** when the agent fails to extract a clear
  answer — show transcript + tel: link, user can try themselves

## Out of scope (for v0.5)

- Any PII handling (member ID, DOB, SSN, address)
- Eligibility / benefits checks (insurer-side, needs PII)
- Scheduling, confirming, rescheduling, canceling appointments
- Live bridge / Mode 2 / human-handoff during call
- Batch calls ("call all 10 of these")
- User accounts / signup / login
- Payment, billing, monetization
- Any vertical other than dental

---

## Why zero-PII

1. **Onboarding stays at 30 seconds.** Three form fields, no signup, no
   profile.
2. **No HIPAA exposure.** The tool never handles PHI, so no BAAs, no
   storage hardening, no audit logs needed.
3. **No representative-of dance.** Agent never claims to act on a
   patient's behalf; it's just asking general questions any prospective
   patient could ask.
4. **Failure cost is low.** Wrong info = user discovers when they call.
   No identity mix-ups, no commitment errors.
5. **It's the actual MVP.** The full vision is v1+. The zero-PII shape
   is a real product on its own.

## User flow (secondary — web form)

This describes the web form at `localhost:8000`, which is the
secondary mobile/demo surface. The primary flow runs inside the
user's LLM host via the MCP server — see "Distribution shape" above.

1. User lands on `/`, sees a form with:
   - Task type (dropdown: plan acceptance / pricing / hours / availability)
   - Dental office phone number
   - Context fields (e.g., insurance name + plan, procedure name) —
     depends on task type
2. Submit → backend places the call.
3. Live transcript view streams turns as they happen.
4. Call ends → results page shows: structured answer + full transcript +
   click-to-dial link to the office.
5. If agent couldn't extract a clear answer → results page says so and
   the click-to-dial is the primary CTA.

---

## Failure escape hatch (not Mode 2)

The agent doesn't bridge the user mid-call. If it can't get a clean
answer, it ends the call and the UI surfaces:

> "We couldn't get a clear answer. Here's what we heard — want to try
> calling yourself?"

with the office's phone as a `tel:` link. Handles the "what if it
fails" worry without real-time orchestration. ~30 min of work vs ~6
hours for a real bridge.

---

## Expansion path (post-v0.5)

Each step ships standalone and adds users who can use the previous step:

- **v0.6** — Batch shopping ("call all 10 of these for me"). Same
  zero-PII tech, just N calls instead of 1.
- **v0.7** — Eligibility check (requires PII upload). First user
  account, first PHI surface.
- **v0.8** — Scheduling via Pattern B (async two-call: scout + commit).
  First Mode 2 feature.
- **v1.0** — Full care manager (verify, schedule, follow up). Real PHI
  infra, BAAs, encryption at rest.

## Done when

A friend who is not Jay can:
1. Install the DialAgent MCP server in their LLM host (paste a JSON
   snippet into Claude Desktop / Cursor / etc., restart) — or use the
   web form at `localhost:8000` as the fallback path
2. In their normal chat, say something like "call my dentist at
   <number> and ask if they take Delta Dental PPO"
3. See the structured answer land in the same chat within ~3 minutes
4. Click-to-dial the office themselves if the agent fell short

Five friends use it. At least two come back unprompted. That's the bar.
